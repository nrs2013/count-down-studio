## B. スマホ連携（Firebase /cds/now）系

| # | 重さ | 内容 | 起きること |
|---|---|---|---|
| B1 | 高 | Wi-Fi 瞬断でサーバー側がデータを自動削除する仕掛け（onDisconnect）はあるのに、**再接続したら書き直す仕掛けがない**。送信は「状態が変わった瞬間」しか走らない | 会場 Wi-Fi が30秒切れる → スマホ全員「CDS WAITING…」→ 次の曲替わりまで復帰しない（長い曲なら5分以上） |
| B2 | 高 | MC/アンコールのカウントアップ情報（経過秒）が Firebase に送られていない（remainingMs=0, totalMs=0 だけ送って終わり） | LED は「MC TIME 03:24」と進むのに、スマホは水色の 0:00 で凍りっぱなし |
| B3 | 中 | タイマー待機中（開演前）のキューは Firebase 送信前に return されて届かない | 開演前の STAND BY! がスマホに出ない |
| B4 | 中 | END サマリー・DOOR OPEN/開演時刻インフォ・サブタイマー（MEMBER IN）はそもそも Firebase に送る設計がない | 終演の瞬間、LED はサマリー表示・スマホは「WAITING…」に落ちる |
| B5 | 低 | 分のゼロ埋めが LED と不一致（スマホ 5:30 / LED 05:30） | 見た目の不統一 |
| B6 | 要確認 | アプリに認証コードが一切ないため、Firebase 側のルールが「誰でも書き込み可」になっている可能性が高い。SCHEDULE STUDIO と同じデータベースなので影響も共通 | URL を知っている人なら誰でもスマホ側の表示を書き換え・消去できる。**Firebase 管理画面でルールを1回確認推奨** |

## C. 編集・リハ中に困る系

| # | 重さ | 内容 |
|---|---|---|
| C1 | 中 | MIDI 受信が編集モード（SET LIST）でもモーダル表示中でも生きている。リハ中にセトリ編集していて卓から信号が来ると、裏でショーが進行。1曲目のノートなら公演集計（開始時刻・MC履歴）が黙って全消しされる。MIDI OFF ボタンは SHOW モードにしか出ない |
| C2 | 中 | 同じ MIDI ノートを2曲に割り当てても警告なし。セトリ上で先にある曲が常に勝つため、終盤でそのノートを叩くと**前の曲へ巻き戻る** |
| C3 | 中 | Excel 一括取り込みの Cmd+Z が1回にまとまらない（60行なら途中状態の控えが約3枚積まれ、1回押すと「半分だけ入った」状態に戻る） |
| C4 | 中 | 曲データ更新（updateSong）が「読み→書き」を別々の処理で行うため、TIME 欄入力→即 X-TIME ボタンのような連続操作で、入力した曲尺が黙って元の値に戻ることがある |
| C5 | 中 | キューの作成・削除は Cmd+Z 対象外なのに Cmd+Z 自体は常時効くため、キュー誤削除→Cmd+Z すると**キューは戻らず、無関係な直前の曲編集が巻き戻る**（二重被害） |
| C6 | 中 | 重複警告で「こちらで使用する」を押したタブは、以後どんな警告にも二度と反応しない恒久フラグが立つ → 2画面同時稼働の温床（既知の宿題「take-over 競合」は未解決で現存） |
| C7 | 中 | デプロイ直後、リロード保留中の旧タブで Excel 取り込みや Firebase 機能を**初めて**使おうとすると部品のロードに失敗する（旧部品がサーバーからもキャッシュからも消えるため。SW の先読み機能は正規表現が実際のビルド形式と合っておらず一件もマッチしない＝全部死にコード） |

## D. 低優先（まとめ）

- END 行が曲番号を1つ消費する（`!isEnd` 抜け、manage.tsx 1789 / performance-editor.tsx 1308）— W アンコ構成で紙のセトリと番号がズレる
- スマホ表示の曲カードに MC/EN/END の見分けがなく、番号も直前の曲と重複表示。END 行も普通に編集できてしまう
- 曲尺に「90」と入れると 1:30 にならず拒否される
- セトリ削除が2段階処理で、途中で落ちると見えないゴミ曲が倉庫に残留
- .scd の検証基準が画面によって違う（編集画面は名前必須、本番画面はファイル名で代用）
- description は書き出すのに読み込まないため、上書き取り込みで混成データになる
- Netlify 専用の意味のないファイル（_redirects）が毎回デプロイされている
- 死にコード：誰も読まない artist 送信、常に null の sectionLabel、送り手のいない "pong" 受信、未使用の skipNextSnapshot / importData / exportSetlist、ModeTabBar の未使用プロパティ2つ
- トースト「4秒で消える」コメントは実際5秒
- Cmd+Shift+R した後のタブは以後の自動リロード対象から外れる（バナーで気づける）
- キュー同キー重複登録が可能（先勝ちで2個目無反応）／押しっぱなし中に削除すると Firebase に死んだキュー ID が残る
